# CLAUDE.md

## Project

Notes — a personal notes app. Users sign up, write notes, organise them into
colour-coded categories, and (optionally) get AI help to categorise/summarise.
Solo build, 7-day hiring challenge. Owner + sole author: **Job**.

## Architecture

- Repo layout: single repo, two apps — `backend/` (Django) + `frontend/` (Next.js). Not a monorepo build system.
- Backend: Python 3.13 · Django 5.2 LTS · DRF 3.17 · SimpleJWT 5.5 (httpOnly cookies) · psycopg 3 · drf-spectacular.
- Frontend: Node 24 · Next.js 16 (App Router) · React 19 · TypeScript strict · Tailwind 4 · TanStack Query 5 · Zod 4.
- Database: SQLite for dev/CI, PostgreSQL for prod parity (`DATABASE_URL`).
- Auth: JWT access + refresh in httpOnly cookies, refresh rotation. `User.USERNAME_FIELD = email` (no username).
- AI assist: Groq via an OpenAI-compatible seam. Degrades to a keyword heuristic when no key is set.
- Full version log: `docs/tech-stack-versions.md`. API contract + data model: `docs/architecture.md`.

## Object-ownership rule (the core security invariant)

A user only ever sees or touches **their own** notes and categories — never another user's.
This is the notes-app analog of tenant isolation, and it is non-negotiable.

- EVERY queryset is filtered to `request.user` (`Model.objects.filter(user=request.user)`), never the full table.
- Never trust an id from the client to imply ownership — scope the queryset, then look up inside it.
- A cross-user access returns **404, not 403** — do not leak that another user's object exists.
- Object writes (`POST`/`PATCH`) set/keep `user = request.user`; the field is read-only in serializers.
- MANDATORY test: user A cannot read/patch/delete user B's note → **404**. Add it whenever a new owned model or endpoint is touched.

## Code style

- **Python:** ruff (format + check, line length 100) is the source of truth. Full type hints on functions.
  DRF serializers validate every request body; never build a response dict by hand where a serializer fits.
  Use `logging`, not `print`. No `# type: ignore` without a written reason.
- **TypeScript:** strict mode, **no `any`** (use `unknown` + narrowing). Validate external data with **Zod at the boundary**
  (API responses, form input, env). Named exports; one component per file with an explicit `Props` type.
- English everywhere. No hardcoded secrets — everything via env (`.env` is gitignored; `.env.example` is the template).
- **No AI traces anywhere.** The human is the sole author. Code, comments, docs, commit messages and PR text must carry
  no AI-assistant attribution (`Co-Authored-By: Claude/Anthropic`, `noreply@anthropic.com`,
  `🤖 Generated with [Claude Code]`, `claude.com/claude-code`, "Generated by Claude"). Strip them on sight. The product's
  own LLM/Groq feature and these policy docs are legitimate references and are fine.

## Testing policy (mandatory)

- Backend: **pytest + pytest-django + factory-boy** (APITestCase / DRF `APIClient`). Frontend: **RTL** (+ Vitest/Jest).
- Every endpoint gets a happy path + an auth-failure path + (for owned models) the ownership-isolation 404 test.
- Tests are deterministic: no real network, no wall-clock/`sleep`, seed any randomness, use factories not fixtures-by-hand.
- **NEVER claim something is "tested" or "works" without actually running it.** Run `ruff check`, `pytest`, `tsc`,
  `npm run lint`/`build`, exercise the path, and show the real output. State exactly what you verified vs. did not run.
  If something cannot be verified here (needs a key, a real DB, a device), say so and ask whether to proceed or postpone.

## Git workflow

- **GitHub Flow:** branch from `main` → PR → review → merge. `main` is always deployable; never push straight to it.
- Branch prefix = commit type: `feat/`, `fix/`, `chore/`, `refactor/`, `docs/`, `test/`, `perf/`, `ci/`.
- **Conventional Commits.** Scopes: `backend, frontend, notes, auth, ai, ci, docker, docs, deps, infra, release`.
- **The human is the sole author. The AI never runs `git commit`/`push`/`merge`/`cherry-pick`/`revert`/`tag` or
  `gh pr`/`release`** — it stages/prepares and hands the exact command(s) as a copy/paste block. Enforced by a PreToolUse
  hook and by the `commit-msg` git hook (`.husky/no-ai-trace.sh`, scans the message only, never staged file content).
- Never commit `.env`, secrets, `node_modules/`, or `.venv/`.

## Commands

- Backend: `cd backend && .venv/bin/python manage.py runserver` · `.venv/bin/pytest` · `ruff check .` · `ruff format .`
  · `.venv/bin/python manage.py makemigrations` / `migrate`.
- Frontend: `cd frontend && npm run dev` · `npm run lint` · `npx tsc --noEmit` · `npm run build` · `npm test`.
- Root tooling: `npm install` (activates husky) · `npm run format` (Prettier). API docs at `/api/docs` (drf-spectacular).

## When unsure

- If 2+ reasonable designs exist, present the trade-offs and let Job decide; write an ADR for anything hard to reverse (`docs/adr/`).
- Never invent realistic-looking PII (emails, names, phone numbers). Never commit secrets, even in test files.

## Active Hooks (do not duplicate their behavior)

- **PreToolUse:** blocks destructive commands (`rm -rf /`, `DROP`/`TRUNCATE`, force-push to main, `git reset --hard`,
  fork bomb, `chmod -R 777`, `curl|sh`); blocks the AI running git commit/push/merge/cherry-pick/revert/tag and gh
  pr/release (hand the command to the human); blocks any command carrying an AI-attribution trace; blocks writing
  secret-ish files (unless `.example`) and content matching API-key patterns.
- **PostToolUse:** on `*.py` runs `ruff format` + `ruff check --fix`; on `*.ts/*.tsx` runs Prettier + ESLint; warns on
  `print(`, `console.log`, `# type: ignore`, `@ts-ignore`, loose `any`. All guarded — no-op if a tool is missing.
- **SessionStart:** prints branch, last commit, pending file count, and Python/Node/Docker versions.
- Full definitions: `.claude/settings.json`.
