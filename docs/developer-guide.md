## 1. Local setup (first 10 minutes)

```bash
# Prereqs: Python 3.13, Node 24 (see .nvmrc), Docker (optional, for prod parity).
git clone <repo> notes-turbo && cd notes-turbo

# Root dev tooling — installs husky, lint-staged, commitlint, prettier, and activates the git hooks.
npm install

# Backend
cd backend
python3 -m venv .venv
./.venv/bin/pip install -r requirements-dev.txt
cp ../.env.example ../.env         # SQLite works out of the box (leave DATABASE_URL empty)
./.venv/bin/python manage.py migrate
./.venv/bin/python manage.py runserver     # http://localhost:8000, API docs at /api/docs

# Frontend (new terminal)
cd frontend && npm ci && npm run dev       # http://localhost:3000

# Verify backend before you start hacking
cd backend && ./.venv/bin/ruff check . && ./.venv/bin/pytest
```

## 2. Git workflow — GitHub Flow

`main` is always deployable. Branch → PR → review → merge. **Never push directly to `main`.**

### Branch naming — prefix = commit type

```
feat/      New functionality              feat/notes-category-filter
fix/       Bug fix                        fix/cross-user-note-404
chore/     Tooling, deps, config          chore/bump-drf-3.17
refactor/  Refactor (no behavior change)  refactor/extract-ai-client
docs/      Documentation                  docs/api-contract
test/      Tests only                     test/ownership-isolation
perf/      Performance                    perf/annotate-note-count
ci/        CI/CD                          ci/add-frontend-build
```

### Conventional Commits

Format: `type(scope): subject`. Scopes: **backend, frontend, notes, auth, ai, ci, docker, docs, deps, infra, release**.

```
feat(notes): add category filter to the list endpoint
fix(auth): return 404 (not 403) on cross-user note access
perf(notes): annotate note_count instead of counting per category
chore(deps): bump djangorestframework to 3.17.1
docs(architecture): document the AI-assist fallback
```

Enforced locally by `commitlint` (`commit-msg` hook) and in CI on PRs. Subject: imperative, lower-case, no trailing period, ≤ 100 chars.

### The human is the sole author

**The AI never runs `git commit` / `push` / `merge` / `cherry-pick` / `revert` / `tag`, nor `gh pr` / `gh release`.** It
prepares the change and hands you the exact command(s) to run. No commit or PR carries an AI-attribution trace
(`Co-Authored-By: Claude/Anthropic`, `noreply@anthropic.com`, `🤖 Generated with [Claude Code]`, `claude.com/claude-code`,
"Generated by Claude"). Enforced by a PreToolUse hook and the `commit-msg` hook (`.husky/no-ai-trace.sh`, which scans the
message only — never staged file content, so the app's own LLM feature and these docs are never blocked).

## 3. Coding conventions

- **Python:** ruff format + check (line length 100) is the source of truth. Full type hints. DRF serializers validate every
  request body. `logging`, not `print`. `snake_case`. No `# type: ignore` without a written reason.
- **TypeScript:** strict, **no `any`** (`unknown` + narrowing). **Zod at every boundary** (API responses, form input, env).
  Named exports; one component per file with an explicit `Props` type. `camelCase`.
- **Ownership everywhere:** every queryset filtered to `request.user`; `user` read-only in serializers, set server-side;
  cross-user access → 404.
- English only. No hardcoded secrets — new env vars go in `.env.example`.

## 4. Definition of Done

Every PR that adds or changes a feature satisfies this before review. If an item does not apply, say why in the PR.

### 4.1 Automated gates (pre-commit + CI)

- [ ] `ruff check .` and `ruff format --check .` green (backend).
- [ ] `npx tsc --noEmit`, `npm run lint`, `npm run build` green (frontend, if touched).
- [ ] `pytest` green (and RTL, if frontend touched) — **actually run, output shown**.
- [ ] `python manage.py makemigrations --check --dry-run` clean.
- [ ] Commits follow Conventional Commits; branch follows §2.

### 4.2 Correctness, security & data

- [ ] **Object ownership**: every queryset filtered to `request.user`; cross-user access returns **404**; an isolation test
      is added/updated for any owned model or endpoint touched.
- [ ] Input validated: **Zod** (frontend) / **DRF serializer** (backend). `user` is read-only and set server-side.
- [ ] Errors use the `{ code, message, details }` shape (custom exception handler).
- [ ] Auth intact: JWT httpOnly cookies, refresh rotation; CORS restricted; CSRF handled for cookie auth.
- [ ] No committed secrets; new env vars in `.env.example`.

### 4.3 Performance

- [ ] No N+1: `select_related`/`prefetch_related` where relations are traversed; `note_count` annotated, not per-row.
- [ ] List endpoints paginated. Query count asserted (`assertNumQueries`) where it matters.

### 4.4 Migrations

- [ ] A migration exists for every model change; destructive changes use an expand/contract plan (documented in the PR).
- [ ] Data migrations (e.g. seeding default categories) are idempotent and reversible.

### 4.5 Docs

- [ ] `docs/architecture.md` API table updated for new/changed endpoints.
- [ ] `docs/tech-stack-versions.md` updated if a dependency was bumped.
- [ ] An ADR added in `docs/adr/` for any decision that is hard to reverse.

## 5. Testing requirements

Stack: **pytest + pytest-django + factory-boy** (DRF `APITestCase`/`APIClient`) for the backend, **RTL** for the frontend.

| Type of change            |  Unit   | API/integration | Ownership-isolation |
| ------------------------- | :-----: | :-------------: | :-----------------: |
| Pure function / helper    |    ✓    |        —        |          —          |
| New/changed endpoint      |    ✓    |        ✓        |          ✓          |
| Owned-model change        |    ✓    |        ✓        |          ✓          |
| Frontend component / flow | ✓ (RTL) |        —        |          —          |

- Every endpoint: happy path + auth-failure (401) + (owned models) the cross-user **404** test.
- Deterministic: no real network (mock the LLM), no `sleep`/wall-clock, seed randomness, factories over hand-built PII.
- **Never claim "tested" without running it.** Show the real `pytest`/RTL output and state what you verified.

## 6. How to open a PR

```bash
git checkout -b feat/notes-category-filter
# ... make changes; the AI stages/prepares but YOU commit ...
git add -A
git commit -m "feat(notes): add category filter to the list endpoint"
git push -u origin feat/notes-category-filter

# Open the PR (fills from .github/pull_request_template.md)
gh pr create --fill
```

CI (backend + frontend + commitlint) must be green. Fill the Definition of Done checklist in the PR body. Squash-merge to
keep history clean, then delete the branch. (Commits, pushes and `gh pr create` are **yours** to run — the AI hands you the
commands, it does not execute them.)
